# HEARTBEAT.md - 心跳巡检任务清单

## 🎯 设计原则
**主动服务，但不干扰**。心跳检查应该：
1. 发现异常立即通知，但不要自动干预
2. 定期整理信息，但保持文件整洁
3. 提供有价值的数据，但不产生信息过载

## 🔄 心跳周期说明
- **默认间隔**: 30分钟 (OpenClaw配置)
- **执行频率**: 每次心跳都会检查本文件
- **时间记录**: 记录每次检查的状态，避免重复

## ⏰ 检查任务分类

### 🔍 每次心跳时执行 (每次)
#### 1. 核心服务健康状态
- **OpenClaw网关**: 检查 `openclaw gateway status` 状态
- **Cron任务**: 检查任务执行状态 (`openclaw cron list`)
- **Discord连接**: 验证频道连接状态
- **语义检索**: 检查memorySearch是否正常工作

#### 2. 系统资源监控
- **磁盘空间**: 监控C盘和G盘使用情况
- **内存使用**: 观察系统内存压力
- **进程状态**: 检查关键进程是否正常运行

#### 3. 异常检测与通知
- **发现异常时**: 立即通过Discord通知，包含具体问题和建议
- **但不干预**: 只报告不自动修复，避免意外操作
- **记录日志**: 异常记录到当天的日志文件中

### 📅 每日执行一次 (00:00-04:00间)
#### 1. 项目待办列表扫描
- **过期任务**: 标记超过3天未更新的任务 (projects.md)
- **优先级调整**: 根据进展调整任务优先级
- **清理完成**: 清理已完成的项目记录

#### 2. 系统性能分析
- **Cron任务统计**: 计算任务成功率、失败率
- **备份验证**: 检查最新的备份是否成功
- **日志轮转**: 清理过期的临时日志文件

#### 3. 用户活跃度检查
- **最近互动**: 检查最近24小时的用户活跃情况
- **待回复消息**: 查看是否有未处理的用户消息
- **上下文准备**: 为可能的互动准备相关上下文

### 📊 每周执行一次 (周一凌晨)
#### 1. 记忆维护任务（每周执行）
检查 `memory/heartbeat-state.json` 中的 `lastMemoryMaintenance` 字段。
如果距今超过 7 天，执行以下维护流程：
1. 读取最近 7 天的日志文件 `memory/YYYY-MM-DD.md`
2. 提炼有长期价值的信息，归档到对应文件：
   - 项目决策和状态 → `memory/projects.md`
   - 问题解决方案 → `memory/lessons.md`
3. 压缩已完成的一次性任务为一行总结
4. 删除完全过期的临时信息
5. 更新 `heartbeat-state.json` 中的 `lastMemoryMaintenance` 为当前日期

#### 2. 长期记忆维护
- **索引优化**: 更新 `MEMORY.md` 索引项
- **过期清理**: 移除超过30天的不重要日志
- **结构优化**: 根据需要调整记忆系统结构

#### 3. 系统健康状况报告
- **性能报告**: 生成每周性能摘要
- **问题趋势**: 分析本周出现的问题类型
- **优化建议**: 提供针对性的优化建议

## 🛠️ 实现细节

### 状态跟踪文件
```json
{
  "lastHeartbeat": "2026-02-27T10:43:00",
  "dailyCheckDone": true,
  "lastDailyCheck": "2026-02-27",
  "weeklyCheckDone": false,
  "lastWeeklyCheck": null,
  "errorCount": 0,
  "lastErrors": [],
  "lastMemoryMaintenance": null
}
```
**文件位置**: `memory/heartbeat-state.json`

### 检查逻辑伪代码
```
每次心跳时:
  1. 读取当前状态
  2. 执行"每次"检查任务
  3. 如果是新的一天且未执行，执行"每日"检查
  4. 如果是新的一周且未执行，执行"每周"检查
  5. 更新状态文件
  6. 如果有异常，发送通知
```

### 通知模板
```markdown
## ⚠️ 心跳检查发现异常

**时间**: {timestamp}
**异常类型**: {type}
**问题描述**: {description}
**影响**: {impact}
**建议操作**: {suggestion}
**完整日志**: {logFile}

状态已记录，等待处理。
```

## 📋 实际操作指南

### 心跳时AI应该做什么
1. **优先检查状态文件** - 避免重复执行
2. **按分类执行检查** - 从最重要的开始
3. **精确记录结果** - 写入对应日期的日志文件
4. **谨慎决定通知** - 只有真正需要关注时才通知

### 避免的问题
1. **不产生噪音** - 频繁的不重要通知会降低价值
2. **不自动修复** - 人工审核后再操作更安全
3. **不累积状态** - 定期清理旧的检查记录
4. **不假设权限** - 只报告能确认的信息

### 心跳 vs Cron 任务选择
| 场景 | 使用心跳 | 使用Cron |
|------|----------|----------|
| 轻量级多任务批处理 | ✅ | ❌ |
| 精确时间触发 | ❌ | ✅ |
| 需要当前会话上下文 | ✅ | ❌ |
| 独立任务执行 | ❌ | ✅ |
| 长时间运行任务 | ❌ | ✅ |
| 定期信息整理 | ✅ | ❌ |

## 📈 质量指标

### 有效性指标
- **异常发现率**: >90% (发现真正的问题)
- **误报率**: <10% (避免假警报)
- **响应时间**: <5分钟 (从异常发生到通知)
- **信息价值**: 每次通知都应包含具体问题和建议

### 性能指标
- **检查耗时**: <30秒 (避免影响主任务)
- **资源占用**: <5% CPU, <50MB内存
- **文件大小**: 状态文件<1KB, 日志<100KB/天

## 🔧 调试与优化

### 问题诊断
1. **心跳不执行**: 检查OpenClaw配置和网关状态
2. **检查失败**: 查看当天的日志文件和状态文件
3. **通知不发送**: 验证Discord配置和频道权限
4. **性能问题**: 优化检查逻辑，减少不必要操作

### 优化建议
- 定期review心跳检查的实用性
- 根据实际使用情况调整检查频率
- 保持通知模板简洁但信息完整
- 记录用户的反馈，持续改进

*心跳检查的目标是让你在需要时获得信息，在你休息时保持警惕，但永远不干扰你的正常工作流程。*
